# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **WeWeb-exported** Vue 3 single-page/multi-page application. The project is **MeetGuru / МитГуру** (`meetguru` in package.json) — an educational platform for kinesiology (courses, articles, clubs, chats, user profiles, seller dashboard/finances). Most of the source is **generated by the WeWeb no-code editor**, not hand-written. Treat it accordingly (see "Working with generated code").

## Commands

```bash
npm run serve      # Vite dev server on http://localhost:8080
npm run build      # Multi-page production build (runs postbuild automatically)
npm run postbuild  # Rewrites per-collection SEO HTML (no-op unless collectionPages configured)

npx eslint .       # Lint (flat config in eslint.config.js; no npm script defined)
npx vitest         # Run tests (vitest is a devDependency; no npm script defined)
npx vitest run src/_common/helpers/urlPath.test.ts   # Run a single test file
```

There is no `test` or `lint` script in `package.json` — invoke the binaries directly with `npx`. `node_modules` may not be installed; run `npm install` first.

## Build model

- **Vite 8 with Rolldown** (`rolldownOptions`, not classic Rollup). Vue plugin, SCSS (`modern-compiler` API), autoprefixer.
- **Multi-page build**: every route is its own HTML entry. The `pages` map in `vite.config.js` (a large generated object) is expanded into `rolldownOptions.input`, each built from `template.html` (Handlebars) with per-page `<meta>`/SEO. Page bodies live in `src/pages/<page-uuid>.js`.
- **Path alias**: `@` → `./src`.
- **Front env vars** are bundled at build time via the `__WW_FRONT_ENV_VARIABLES__` define, sourced from `.env.staging` and `.env.production`. At runtime `src/helpers/frontEnv.js` picks staging vs production from `window.location.origin` (hostnames containing `-staging.` → staging). Do not assume `import.meta.env` alone drives environment selection.

## Architecture

**Entry & bootstrap** — `src/_front/main.js` creates the Vue app, installs both the Vuex store and Pinia, exposes `window.wwLib` and `window._` (lodash) and `window.axios`, registers WeWeb plugins, then calls `wwLib.initFront({ store, router })` before mounting `#app`.

**Routing** — `src/_front/router.js`. Routes are generated from `window.wwg_designInfo.pages` (defined inline in that file). Page components are lazy-loaded from `src/pages/<page-uuid>.js`. Page access rules (`security.accessRule`, roles) come from the same design info.

**`wwLib` runtime** (`src/wwLib/`) — the global WeWeb runtime object. `src/wwLib/services/` holds the service modules: `wwVariable`, `wwFormula`, `wwWebsiteData`, `wwAuth`, `wwCollection`, `wwWorkflow`, `wwPluginHelper`, etc. Front code and generated components reach these through the global `wwLib` (declared `readonly` in eslint + typed `any` in `src/global.d.ts`).

**Two state systems coexist:**
- **Vuex** (`src/store/`) — front runtime state, modules: `websiteData`, `libraries`, `data`, `front`. Accessed as `wwLib.$store`.
- **Pinia** (`src/pinia/`) — editor/"back" state: `backAuth`, `backTables`, `backTableViews`, `variables`, `integrations`, `hooks`, `envVariables`, etc.

**Plugins** (`src/components/plugins/plugin-<uuid>/`) — four WeWeb plugins configured in `plugins-settings.json` and registered in `src/wwLib/index.js`:
- Supabase + Supabase Auth (project at `https://sb.meetgu.ru`; realtime tables listed in `plugins-settings.json`)
- Date (dayjs)
- REST API

**Backend workflows** (`src/_common/helpers/code/`) — WeWeb's server-side workflow invocation layer. `serverClient.ts` (`window.wwServerClient`) is a `@better-fetch` client hitting `${window.location.origin}/api` with a **concurrency cap of 4** and automatic retry on HTTP 429 (up to 20 attempts). `backendWorkflows.js` builds payloads (incl. multipart file uploads) and drives requests. This is the primary path for authenticated server calls.

**`src/_common/` vs `src/_front/`** — `_common` holds editor-and-runtime-shared logic (workflows, formulas, configuration, data init, url/path parsing) and is where most **hand-written, tested** code lives. `_front` holds the runtime app shell, router, and the `wwElement`/`wwLayout`/`wwSection` rendering components.

## Working with generated code

- Regions wrapped in `/* wwFront:start */ … /* wwFront:end */` are WeWeb-managed. Editing them risks being overwritten on the next export.
- **UUID-named files are generated**: `src/pages/<uuid>.js`, `src/components/elements/element-<uuid>/`, `src/components/sections/section-<uuid>/`, `src/components/plugins/plugin-<uuid>/`. Prefer changing hand-written helpers over these unless the task specifically requires it.
- The `.test.ts`/`.test.js` files (concentrated in `src/_common/helpers/`) mark the hand-authored, safe-to-modify surface. Add tests alongside them.

## Deployment convention

Commits are release bumps named `vNNN - <note>` (see `git log`). `window.wwg_cacheVersion` (currently 271) and the `_wwcv=` cache-busting query param track the same version and are updated by the WeWeb export/deploy, not by hand.
